01 — Demo 幻觉

为什么 demo 几乎总是成功？

因为 demo 是一个被精心构造的世界。它通常只有干净数据、短路径任务和几乎为零的失败成本。Agent 在这种环境里不是在面对现实复杂度，而是在解一道被人提前整理好的题。

真实断裂：现实输入来自 PDF、邮件、口语与冲突信息，任务会跨越 20 多步、伴随分支与回滚，而且每次调用都可能产生真实成本与不可逆副作用。

所以 demo 的成功，更多证明的是“这个系统在温室里可运行”，而不是“它在生产环境中可稳定”。

02 — 关键转折

Agent 不是 API，而是长时间运行的动态系统

API 的世界是输入、输出、一次调用、无状态。Agent 的世界是多步执行、持续状态、与环境交互，以及带有副作用的不可逆行为。

Intent → Plan → Action → Observation → Feedback ↺ 控制循环

一旦把 Agent 看成系统，核心问题就从“它会不会做”变成“它稳不稳定”。稳定系统至少需要状态可控、反馈有效、噪声可抑制、目标不漂移，而当前许多 Agent 恰恰在这四点上同时失分。

03 — 失稳模式

四种典型的失稳方式

  • 01
    循环失稳 系统不断重复动作却没有新信息进入，成本持续上涨，本质是缺少“进度信号”，不知道自己是否在靠近目标。
  • 02
    语境漂移 新信息覆盖旧约束，初始目标逐渐丢失，系统行为越来越偏，本质是状态没有被压缩、锚定与重述。
  • 03
    工具级联失效 一个工具调用出错，后续步骤继续基于错误前提运行，问题会一路扩散，本质是缺少验证和回滚机制。
  • 04
    成本失控 Token、时延、调用次数暴涨，但结果并未改善，本质是没有资源预算与停止条件。
统一解释：这些表象背后其实是一件事，Agent 系统缺少足够的控制论结构。它有模糊目标和很强执行，却没有足够强的反馈控制。
04 — 隐蔽失败

为什么模型越强，问题反而越难发现？

因为强模型更擅长“优雅地失败”。它能给出完整推理链、合理措辞和看似自洽的解释，让人误以为系统是对的。比报错更危险的是，它开始在错误之上维持一种表面的可信度。

这也是很多团队误判问题来源的原因。他们以为要换更强模型、调更细 prompt、喂更多数据，但真正的缺口不在模型层，而在架构层。

05 — 三篇闭环

从意图到语境，再到系统失稳

前面的链条到这里已经连起来了：意图表达不清，context 缺乏结构，最终就会在执行阶段表现为系统失稳。这不是三个独立问题，而是一条连续的因果链。

  • 篇02
    意图不清 目标和约束无法稳定传递给系统。
  • 篇03
    语境无结构 状态与背景信息无法被持续、可靠地组织。
  • 篇04
    系统必然失稳 执行层就会出现循环、漂移、级联错误与预算爆炸。
收束：真正的问题不是 Agent 能不能做这件事，而是它能不能在第 100 次还做对。

更深的问题则是：如果控制结构本身也被外包给另一个 Agent，系统还是否可控？

Agent 的问题从来不只是“还不够强”，
而是我们还没有把它当作系统来设计。
