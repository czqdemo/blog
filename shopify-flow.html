用自然语言生成 Flow 自动化工作流：
Shopify 的 Agent 微调方法论

Shopify 将 Qwen3-32B 微调成一个工具调用 Agent，让商家用一句话就能生成复杂的自动化流程。比原来的前沿闭源模型快 2.2 倍、成本降低 68%，还跑赢了 GPT-4 级别的对手。

如果你只是在闭源模型 API 上叠需求，拥有 API key 的竞争对手随时能复制你的产品。真正的差异化来自三件事：专有数据、训练方案，以及持续迭代的速度。

Shopify 拥有大多数公司没有的优势——一个有数百万商家实时交互的产品表面，这些交互直接反映模型输出的质量好坏。这个反馈闭环是差异化的基础，但前提是你要不断地从中学习。

这篇文章介绍 Shopify 如何将 Qwen3-32B 微调成一个工具调用 Agent，用于 Sidekick（Shopify 的 AI 商业助手），把商家的自然语言输入转化成 Shopify Flow 自动化工作流。过程中遇到的每一个坑，都比论文里描述的更狡猾。

一、构建训练数据集

Shopify Flow 是一个自动化平台，商家可以用触发器、条件和动作来搭建工作流。对非技术商家来说，从空白画布开始构建正确的工作流是令人望而却步的——Sidekick 的任务是让他们说一句话就搞定。

冷启动问题

微调需要训练数据，但这个功能还没上线，所以根本没有生产对话可以学习。Shopify 的解法是：从现有生产工作流反向推导用户意图。

数千名匿名商家已经在 Flow 里手动搭建好了工作流。团队对这些工作流进行采样和质量过滤：过去七天内至少运行过一次、来自拥有两个以上合格工作流的商家、每个描述符只取一个样本以确保多样性。

有了这批高质量工作流之后，流程是这样的：

  1. 从生产环境中抽取一个经过验证的热门工作流
  2. 用更强的 LLM 生成一条"能引出这个工作流"的自然语言请求
  3. 构建完整的多轮工具调用轨迹——一个理想 Agent 从头到尾应该怎么执行。这一步是工程量最大的部分。

用这套合成数据集微调 Qwen3-32B，并以 300 个手工制作的样本作为 benchmark 进行评估（覆盖 Flow 的主要使用场景）。评估框架用 LLM judge 进行语义正确性对比，用程序化方法验证语法正确性。

语义 LLM judge 对生成工作流与期望工作流进行语义对比
语法 程序化检测：条件格式、引用、配置是否合法
延迟 从请求到工作流交付的耗时
工程洞见
如果还没有用户交互数据，就从用户已经产出的输出物（工作流、文档、配置文件）出发，逆向推导意图。这往往是在指标还没跑起来之前，最正确的第一步。

二、Python DSL：让模型在它熟悉的语言里学习

Shopify Flow 工作流在内部用一套 JSON-based DSL 表示，专为后端解析、校验和执行设计。这个格式对生产系统是理想的，但对 LLM 来说很不友好——程序逻辑被深埋在深层嵌套的 JSON 结构里，而这种写法在预训练数据中极为罕见。

团队的解法是：不强迫模型从零学习 Flow 的原生格式，而是把任务重新用更接近模型训练分布的表示方式来描述——让模型用 Python 来写工作流。

一个 Transpiler（转译器）负责将 JSON DSL 转换成语义等价的 Python：

Python DSL 示例（模型输出格式）
# 当订单金额超过 $100 时，自动打标签 "VIP"
@trigger("order/created")
def tag_vip_order(order):
    if order.total_price > 100:
        order.add_tag("VIP")

这是同一个工作流，同样的语义，但模型现在生成的是 Python 而不是数据格式。Python 在预训练数据里占有大量比重——装饰器、if/else、变量、循环、函数调用，这些都是模型已经熟悉的语言模式。

指标 JSON DSL（原格式） Python DSL（新格式）
语法正确率 基准 +22 个百分点
语义正确率 基准 +13 个百分点
模型学习难度 需同时学格式和任务 只需专注于学任务本身
商家是否看到 Python — 不可见，由后端静默转换
核心原则
将目标格式从"分布外"移到"分布内"，把问题从「既要学新语言、又要学任务」变成「只学任务本身」。

这套方案的保障来自严格的端到端 Transpiler 测试：所有经由 Sidekick 创建的生产工作流都会被持续 round-trip 测试（JSON → Python → JSON），任何不一致都会在进入训练数据之前被拦截。推理时，模型写 Python，Transpiler 转换成 JSON 送往 Flow 后端，商家和后端都不需要知道 Python 的存在。

三、镜像生产环境：细节里的恶魔

表示方式只是数据问题的一半。另一半是确保训练数据与生产环境完全一致——而团队发现，模型对这种一致性有多敏感，超出了任何人的预期。

每一处差异被弥合，无论看起来多微小，都能提升评估分数：

工具命名与排序

训练数据里用的是完整带前缀的名字 flow_app_agent_task_search，但推理时同一个工具叫做 task_search。功能完全相同，但模型把它们当成了两个不同的工具。去掉前缀以匹配推理环境，准确率提升。工具在系统提示中出现的顺序也很重要——训练和服务时的顺序不同，性能就会下降。

工具响应格式

工具响应返回的是含多个字段的 JSON 对象。训练数据里 key 按字母顺序排列，如果生产环境返回时顺序不同或多了一个字段，模型会注意到。训练数据展示的内容和生产 API 实际返回的内容之间的任何漂移，都会降低准确率。

系统提示与工具描述

生产环境中工具描述会随产品团队的迭代频繁更新。每次更新都必须同步到训练数据，否则模型行为就会偏移。保持两者同步是持续的工作，不是一次性的。

警告
模型将每一个 token 都当成信号，不管你是否有意为之。格式细节不是可以忽略的噪声——它们是模型学习的一部分。

四、优化工具调用栈，压缩上下文

Agent 每调用一次工具，其响应就会进入上下文。上下文增长，延迟增长，成本增长。更糟糕的是，无关上下文会稀释信号——模型在处理它用不上的信息时，推理准确率会下降。

团队对工具接口进行了重构，最小化每一步的上下文：工具先返回轻量摘要，模型扫描摘要，选出需要的内容，再仅对那些条目检索完整信息。两次廉价调用代替一次昂贵调用。

例如，Flow 有数百个可用的触发器、条件和动作。一次搜索可能匹配 100 个结果。不是把每一个的完整配置 Schema 都塞进上下文，task_search 只返回名称和描述，模型选出真正需要的 2-3 个，再调用 task_configuration 获取完整 Schema。上下文保持精简，推理保持聚焦。

五、让训练跑得足够快

数据管道规模扩大后，一个矛盾随之出现：更多训练数据能提升准确率，但每次训练也会变慢；训练越慢，迭代次数越少，改进越慢。团队的解法是：在基础设施上做投资，让两者都可以兼得。

Qwen3-32B 在两个节点的 H200 GPU 上使用 FSDP（全参数分片数据并行）训练，完整训练一次只需 12 小时，足以支撑每周重训，中间还能穿插多次实验性运行。

整个管道——数据采集、训练、评估、部署——运行在 Tangle（Shopify 开源的 ML 实验平台）之上。Tangle 将每个步骤编排成一个可复现的工作流，并带有智能缓存：某个部分发生变化时，只重新运行受影响的步骤。

CometML 跟踪每次运行，HuggingFace 托管数据集和 Checkpoint，CentML 负责生产环境推理服务。每周重训无需人工干预自动运行。

六、Benchmark 不等于现实——那 35% 的差距

合成数据让模型在离线 Benchmark 上达到了与闭源模型持平的水准。所有追踪指标都显示：微调模型已经准备好上线了。于是他们把它部署到 1% 的流量上，看看实际表现。

在 1% 流量下，微调模型的工作流激活率（商家实际开启 Sidekick 生成的工作流的比率）比 Prompt-based Agent 低了 35%。

Benchmark 覆盖了团队预期商家会问什么，但没有覆盖他们实际上问什么——编辑现有工作流、处理邮件配置、接入第三方集成、询问 Flow 但并不打算创建工作流。

模型在分布内表现良好，但真实流量很快暴露出合成数据没有覆盖的分布外请求。低流量的早期部署精准告诉了团队下一步该专注什么。

评估策略
工作流激活率是第一个生产信号，但它反映的是商家行为，而不是模型质量。团队最终以领域专家校准的 LLM judge 为优化目标，同时把激活率作为不可退步的护栏指标。

七、持续飞轮：从追赶到领先

诊断层：找到差距在哪

团队构建了一个 LLM judge 对每次对话从多个维度评分：助手是否正确理解了商家意图、是否只在合适时才给出 Flow 方案、是否选对了组件、后续步骤是否清晰。Judge 对每个维度分别评分，而不是只给一个通过/失败的结论。

为了校准 Judge，团队收集了数百个对话的人工标注，反复调整直到其评分与人类判断对齐，再对照生产激活率进行验证。

一套标签系统沿多个维度对每个工作流进行分类：使用了哪些触发器、检查了什么条件、调用了哪些动作、是否涉及第三方集成。对比各标签切片的性能，精确定位模型在哪里表现差。某个切片性能下降时，就知道该补充什么类型的数据。

诊断结果指向了具体的修复方向：

  • 邮件工作流占失败案例的 25%，针对性补充邮件示例
  • 多样化的条件模式占 16%
  • 工作流编辑（合成数据从未覆盖的场景）

飞轮：持续运转的改进循环

关闭差距只是第一个测试，真正的目标是保持领先。每一条生产对话都变成了训练信号。团队对高质量样本进行采样：商家在对话后实际激活了工作流的对话。Judge 打分，高分对话自动进入训练池，低分对话隔离等待人工审查。

1
摄取生产对话
↓
2
LLM judge 评分
↓
3
高质量样本进入训练池；低质量隔离审查
↓
4
通过标签切片分析识别差距
↓
5

系统随生产流量的变化自动改进，让团队可以把精力放在扩展覆盖范围和修复系统性差距上，而不是手动整理数据。

八、下一步：模拟环境与在线学习

飞轮已经运转，但闭源模型和自研模型之间的赛跑不会停歇。每隔几个月，新的前沿模型就会拉高门槛。唯一的应对方式是持续复利：更好的数据、更好的训练、更好的评估、更快的迭代。

模拟环境

一个沙箱，让模型可以生成工作流并接收结构化反馈（是否会成功执行），而不影响真实商家。模型在模拟 Flow 环境中编写测试用例并运行，创造出可验证奖励（Verifiable Rewards）的条件。这为从更强教师模型的蒸馏以及在线策略优化打开了大门。

从离线学习到在线学习

目前的一切都是离线学习（Off-policy）：模型从事后精心筛选的示例中学习。有了模拟环境提供的可验证奖励，下一步是策略优化——让模型从自己生成的轨迹中学习。目标是一个能发现更好策略的模型，而不只是复现它见过的东西。

评估自动校准

今天的 LLM judge 是依靠人工标注和生产激活率手动校准的。但商家行为会变，新的集成会上线，新的工作流模式出现的速度比手动重新校准要快得多。让 judge 基于实时生产信号自动校准，是下一个评估挑战。

九、这套方法适用于哪些场景？

Shopify 的经验可以在以下情形下推广：

  1. 任务需要工具调用：模型必须推理、行动并整合外部结果，而不只是生成文本。
  2. 输出格式是自定义 DSL：这种 DSL 不在预训练数据中，但其语义可以用模型熟悉的语言来表达。
  3. 端到端 Transpiler 是可行的：可以在分布内表示与生产格式之间双向转换。
  4. 有生产反馈闭环：合成数据让你起步，但真实世界数据才能带你到生产质量。

结语

这套微调 Flow Agent 的系统现在服务于 Shopify 大部分的生产流量。

没有哪个单一技术让这一切成为可能。每个阶段都建立在上一个阶段之上：合成数据生成需要 Python DSL 来弥合准确率差距；Python DSL 需要生产镜像才能在真实环境中撑住；生产镜像需要稳定到可以信任的基础设施；而当 Benchmark 说"准备好了"但生产说"不行"时，飞轮在两周内关闭了这个差距。

六个月前，这套系统跑在一个我们控制不了的前沿模型上。现在，它跑在我们训练的模型上，在我们拥有的基础设施上，用只有我们才有的数据不断改进，成本降低了 68%。正在运行的这个版本，已经比正在它身后重新训练的下一个版本要差了。我们曾经租用别人的土地。这是我们开垦自己土地的第一英里。
