语境的重量
Context 不是参数，是认知基础设施

很多人把 context 理解成“窗口大小”，以为 1M tokens 就意味着更聪明。但工程现实更接近：世界变大了，AI 更迷路了。真正的分水岭不在窗口长度，而在语境的筛选、组织、更新与遗忘——也就是 context system。

01 — 一个词

Context 被误解了：不是“窗口”，而是“工作状态”

大多数人听到 “context”，脑子里是 32K、128K、1M tokens，或者“能不能把整个代码库塞进去”。产品也在这么卖：我们支持 1M context window。

听起来像是世界变大了，AI 更聪明了。但现实更接近：世界变大了，AI 更迷路了。

一个致命事实：模型的“记忆”不是记忆，更像滑动窗口。超出窗口就不存在；在窗口中间权重下降；信息越多注意力越稀释。

在长上下文中，信息关注是不均匀的，中间部分更容易被“忽略”。甚至更极端：上下文越长，推理质量可能反而下降，因为信噪比在降低。

02 — 退化

Context Rot：语境腐化不是 bug，是系统性退化

你可能已经见过：Agent 一开始很聪明，跑久了开始跑偏，忘记最初目标，做出奇怪决策。这不是偶然，也不只是实现细节问题。

Context Rot：新信息不断进入，旧信息没有被压缩或淘汰，噪声逐渐淹没信号。交互越长，context 噪声越累积，推理能力越下降。

很多人以为问题是 context 不够大，但更本质的原因是：context 没有结构。

把整个代码库、所有日志、所有对话一次性塞给模型，你以为给了“更多信息”，系统视角看却是信息熵爆炸：不相关信息、过时信息、冲突信息混在一起，模型面对的不是更清晰的世界，而是更混乱的输入。

03 — 定义

关键反转：Context 不是“多”，而是“对”

我们换一个定义：Context = 可被模型利用的“有组织信息状态”。它不是原始数据，不是全量输入，也不是历史记录本身。

它是一个经过选择、压缩、排序、结构化的信息子集。

这意味着：更大窗口并不能自动生成更好的语境，反而可能降低系统的信噪比。

行业也在形成共识：扩大窗口 ≠ 解决问题。原因往往来自三件事：注意力不是线性的；成本与延迟爆炸；更严重的是信噪比下降——工具输出、中间推理、错误尝试全部进入 context，模型被“自己生成的垃圾”淹没。

04 — 结构图

真正的难点在前两步：采样与组织

把 context 当作系统设计问题，你会发现核心断点不在 LLM，而在“世界状态如何被采样与组织”为可用的语境。

Raw World State → Selected Context → Structured Context → Token Sequence → Latent Reasoning ↺ 循环
关键断点：大多数系统只做了最后两步，但真正的难点在前两步：如何采样、如何组织。
05 — 范式迁移

从 Prompt Engineering 到 Context Engineering

这是一种范式迁移：从写 prompt，变成设计 context；从调模型，变成设计信息流；从优化输出，变成优化认知路径。

Context Engineering 关心的是：信息从哪里来、如何筛选、如何更新、如何遗忘。

一个更狠的视角：LLM 是推理引擎，Context 是工作记忆，Memory system 是长期记忆。没有 context，LLM 就像一个没有短期记忆的人类。

和上一篇连接也很关键：意图定义“做什么”，context 决定“怎么理解这个做什么”。没有 context，意图无法被执行。

06 — 分裂

一个危险但真实的趋势：系统会分裂成两类

未来的系统可能分裂成两类：一种是模型驱动，依赖更大模型、更大窗口、更强推理；另一种是 context 驱动，投入 memory、context 管理与信息结构。

  • A
    模型驱动 更大模型、更大 context、更强推理。短期有效，但长期不稳定，语境腐化风险更高。
  • B
    Context 驱动 强 memory 系统、强 context 管理、强信息结构，更接近“可持续智能”。
07 — 收束

你在构建什么样的“世界”？

当你在做一个 AI 系统时，真正的问题不是模型选哪个，而是：你在给它构建什么样的世界？

留一个裂缝：如果 context 是认知基础设施，会出现一个更深的问题：谁控制 context，就在控制“AI 看见的世界”。如果 AI 的“视野”被设计过，它的“判断”还算客观吗？

如果说上一篇让人意识到“你不会表达意图”，这一篇要让人意识到：就算你会表达，没有 context 结构，一切依然是噪声。

更大的窗口只是更大的输入，
更好的语境 才是更可靠的认知系统。
